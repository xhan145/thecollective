 Achievement Badge System
 Developer handoff for meaningful, behavior-based achievements inside Collective.

3. Badge Page UI Direction
This page should feel like a calm progress archive, not a trophy wall. The user should understand what they earned, why
they earned it, and what meaningful action unlocks the next badge.
Surface
Implementation requirement
Badge Page
Grid of earned and locked badges with category filters: All, Core Loop, Practice, Proof, Feedback,
Trust, Communication, Contribution.
Profile
Show only 6 selected identity badges. Do not show every badge by default.
Proof Detail
Show badges only when relevant to the proof, such as Clear Proof, Before and After, Evidence of
Growth.
Unlock Modal
Use subtle progress motion and one sentence explaining the behavior that earned the badge. Avoid
confetti overload.
Locked State
Show badge silhouette, name, and one clear next action. Do not reveal hidden badges until unlocked.

6. Developer Implementation Spec
Treat achievements as a rule-driven system backed by user events. Avoid hardcoding checks into screens. Screens should
emit events; the achievement service should evaluate unlocks.
Recommended Data Model
-- Core achievement definition
create table achievements (
  id uuid primary key default gen_random_uuid(),
  slug text unique not null,
  name text not null,
  description text not null,
  category text not null,
  stage text not null,
  rarity text not null default 'Common',
  icon_url text,
  xp_reward int not null default 0,
  trust_reward int not null default 0,
  is_hidden boolean not null default false,
  is_active boolean not null default true,
  unlock_event text not null,
  unlock_condition jsonb not null default '{}',
  created_at timestamptz not null default now()
);
-- User achievement state
create table user_achievements (
  user_id uuid not null references auth.users(id) on delete cascade,
  achievement_id uuid not null references achievements(id) on delete cascade,
  unlocked_at timestamptz not null default now(),
  source_event_id uuid,
  progress_snapshot jsonb not null default '{}',
  primary key (user_id, achievement_id)
);
-- Generic activity/event ledger
create table user_events (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  event_type text not null,
  entity_type text,
  entity_id uuid,
  metadata jsonb not null default '{}',
  created_at timestamptz not null default now()
);
create index user_events_user_type_idx on user_events(user_id, event_type);
create index achievements_unlock_event_idx on achievements(unlock_event);
Event Evaluation Flow
onUserEvent(event):
  insert into user_events
  candidateAchievements = achievements where unlock_event = event.type and is_active = true
  for achievement in candidateAchievements:
    if user already has achievement: continue
    progress = buildProgressSnapshot(user_id, achievement, event)
    if evaluate(achievement.unlock_condition, progress):
      insert user_achievements
      apply xp_reward
      apply trust_reward if positive
      create notification: achievement_unlocked

Build Order
1. Add event ledger and achievement tables.
2. Seed MVP 30 achievement definitions.
3. Emit events from Direction, Practice, Proof, Feedback, Reflection, and Trust flows.
4. Implement server-side or edge-function achievement evaluator.
5. Build Badge Page grid and Badge Detail sheet.
6. Add achievement unlock notification and profile badge picker.
7. Add analytics for unlock rate and drop-off between badges.
Definition of Done
All MVP badges can be unlocked from real app behavior, not manual toggles.
A user cannot unlock the same badge twice.
Locked badges show the next meaningful action unless hidden.
Badge visuals render correctly in grid, profile, unlock modal, and proof detail contexts.
Achievement checks run outside UI components so they work across web/mobile clients.
Trust rewards are small and auditable; high-trust badges may require moderation review later.
